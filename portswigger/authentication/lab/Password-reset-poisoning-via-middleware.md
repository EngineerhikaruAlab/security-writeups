# Lab: Password reset poisoning via middleware

## 概要
このLabではパスワードリセット時にメールアドレスに送るリセット専用のURL発行の脆弱性を扱う。
具体的には攻撃者が有効なアカウント名でサーバーにリセット用のURLを送るよう指示し，そのときHTTPリクエストの
HOSTヘッダーを自分の悪意あるサイトにすることでサーバーがそのサイトのドメインに犠牲者アカウントとのトークンを加えて
犠牲者アカウントに送る．そして犠牲者がそのリンクを踏めば，ハッカーのサイトのサーバーのアクセスログに秘密のトークンが
ばれる．
目的はこの脆弱性を理解し，Burpで攻撃することである。

---

## 脆弱性の種類
Authentication

---

## 攻撃の結論（最短）
パスワードリセットHTTPリクエストにユーザー名を対象アカウントにし，X-Forwarded-Hostに自分のサーバーのURLとすることで
対象者に本物のサーバーが作った，秘密のトークンを含んだURLを踏ませ，自分のサーバーのログに残し，トークンを見て，パスワードリセットした．

### 2. 仮説

---

### 3. 検証
-まず自分のアカウントでパスワードリセットを試し，HTTPリクエストとリセット挙動を確認
- Burpで実施した内容：
  - Repeaterで自分のアカウントのリセットリクエストのusernameパラメータを対象アカウント，新たにサーバーをだますように
  - X-Forwarded-Hostヘッダーをつけて自分のサイトのサーバー向けにした．（対象がリンクを踏めば自分のサーバーのログに残るように）
  - 分かったトークンをつかってリセット用のPOSTリクエストのトークンをこれに書き換え，送信した．
  - そのあと対象アカウントでリセットしたパスワードをいれた．
  
- 結果：
  - HTTPリクエストにRefererヘッダーがあるがこれはどのアカウントでも同じでホストヘッダーかRefererからのURLと
  - 一時的にリセット用に生成したトークンをいれてリンクを生成している．
  - 自分のサーバーに生成されたトークンとサーバーのURLの組み合わされたGETリクエストがログに残った．
  - ログインできた．

## 学んだこと
- （このLabで理解できたこと）
ハッカーは必要な情報今回はトークンに狙いを定め，リセット用のURL生成ルールを巧みに利用し，本物のサーバーが対象者に
リセット用のリンクを踏ませて，自分のサーバーにアクセスさせトークンの情報をみるという非常に考えられた方法を学んだ．
ここで大事なのはセキュリティにかかわる技術（URLの生成など）を理解し，ここはこうしたらどうなるかを常に考えること，
そうすれば脆弱性を利用した攻撃が思いつき，それを防げる．
また自分のアカウントでこのサーバーはどんな仕組みなのかを入念に確かめて仮説を立てるのが一番大事
