# Lab: URL-based access control can be circumvented

## 概要
このLabではプラットフォーム層（Nginxやロードバランサー）でユーザーグループの特定のHTTPリクエストメソッドや
URLリクエストをはじいたあとのフレームワーク（Laravelなど）が介在することによる脆弱性を扱う。  
目的は脆弱性を理解し、Burpで確かめ、
HTTP Header-based Access Control Bypass（HTTPヘッダーによるアクセス制御の回避）を学ぶことである。

---

## 脆弱性の種類
Access Control

---

## 攻撃の結論（最短）
フレームワークの脆弱性をついて、X-Origin-URLヘッダーに/adminを書き、リクエストラインにしたいURLのクエリを書いて送信した。

## 学んだこと
- （このLabで理解できたこと）
- プラットフォーム層（NginxやWAF）がガードしても内部のフレームワークが新しく道を作って通したらなんの意味もない。
- フロントエンドのプロキシ（門番）」と「バックエンドのアプリケーション（受付）」で、リクエストの解釈ルールが違うため脆弱性が起きた。

## メモ（任意）
- 今回のフレームワークの設定をだますためのX-Original-URLは純粋なパスを上書きするためのものでURLのクエリを書いても
- パスとしてしか認識されない。
- 攻撃者はX-Original-URL、X-Rewrite-URL、X-Forwarded-For、X-Host など、バックエンドのフレームワークが「気を利かせて上書き処理をしてくれそうな隠しヘッダー」を手当たり次第に付与し、手前の制限をすり抜けられないか（Bypass）を必ず検証する。
- 対策としてNginxなどの設定でX-Original-URLなどの特殊なヘッダーを無効化するなどがある。
- この脆弱性はアプリ開発者がインフラの設定を知らないことでよく起きる。
