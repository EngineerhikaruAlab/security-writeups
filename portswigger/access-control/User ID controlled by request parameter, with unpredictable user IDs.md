# Lab: User ID controlled by request parameter, with unpredictable user IDs

## 概要
このLabではURLのidパラメータを予測できないようにしたGUIDに対するアクセス制御の脆弱性を扱う。  
目的はここの脆弱性（バックエンドのアクセス制御の怠慢）を理解し、攻撃手法を学ぶことである。

---

## 脆弱性の種類
Access Control

---

## 攻撃の結論（最短）
- 対象ユーザーのGUIDをプロフィールから入手し、自分のアカウントのAPIキーのGETリクエストのクエリにそのGUIDをはってリクエストした。

---

## 攻撃の流れ

### 1. 観察
- （画面を見てわかったこと）
対象ユーザーのプロフィールがあった。

---

### 2. 仮説
- （どんな脆弱性だと考えたか）
- 対象ユーザーのGUIDは公開されているのにもしかしたらバックエンドでアクセス制御を行っていない可能性がある。
---

### 3. 検証
- Burpで実施した内容：
  - 対象ユーザーのアカウントのプロフィールを見てGUIDをBurpで確認
  - 有効なアカウントのAPIのGETリクエストのクエリをそのGUIDに書き換えてリクエスト
  - （例：有効なユーザー名を特定）
  - （例：その後パスワード候補を `Intruder` で試行）
- 結果：
  - APIキーを入手できた。どうやらバックエンドできちんとアクセス制御をしていない。

- システム上どうしても必要な正常な動作である。
- GUIDをHTMLのソースコードから見つけ出したこと自体は、「情報漏洩（セキュリティ事故）」ではなく、単なる「公開情報の確認」であり、
- 真の脆弱性（異常）は、**「推測しにくい名前（GUID）を採用したことで安心してしまい、バックエンドのプログラムで
