# Lab: User role can be modified in user profile

## 概要
このLabではユーザーの自己申告を信じてアクセス制御をサーバーで行わない脆弱性を扱う。  
目的は脆弱性を理解し、実際にBurpで確かめ、体感することである。

---

## 脆弱性の種類
 Access Control

---

## 攻撃の結論（最短）
- 有効なアカウントでログインし、update Emailリクエストでroleidをjson形式で変えるようにし、そのあと/adminに
  アクセスし対象のアカウントを削除した。
---

## 攻撃の流れ

### 1. 観察
- （画面を見てわかったこと）
有効なアカウントがある。

---

### 2. 仮説
- （どんな脆弱性だと考えたか）
  ユーザーがパラメータを変えてそれを信じてadminのアクセスを許す脆弱性だと考えた。
- （どこをどう試せば検証できるか）
  有効なアカウントでログインし、管理者パネルに行くか判定されるパラメータを探す。

---

### 3. 検証
- Burpで実施した内容：
  ログインしたあとにCookieを確認
  次にUpdate Emailでどのようなレスポンスが帰ってくるか見た。
  このUpdate Emailリクエストでjson形式にroleidを2として紛れ込ませ送った。

- 結果：
  roleidはなかった。
  json形式でemailとroleidを含んだ情報がレスポンスでやってきた。
  レスポンスでroleidが2となって返ってきた。 

---

### 4. 成功条件・根拠
- 成功したと判断した根拠：
  レスポンスでroleidが2となって返ってきた。
---

## 使用ツール
- Burp Suite（`Proxy` / `Repeater`）

---

## 学んだこと
- （このLabで理解できたこと）
  この脆弱性はとくにマスアサイメントと呼ばれる有名な脆弱性で開発者の手抜きで起きるもの
  例えば、入力項目は個別に変更するか判定し、個別に変えるべきだが、横着して
  送られたデータをそのままforeach文で上書きしている。
  また、変えた後のレスポンスにサーバーの機密の入力項目の結果を返してしまっている。
  攻撃者はレスポンスがうっかり逃した変数名や値を見逃さない.
- （次回以降に活かせるポイント）
  サイトにある問題に関連すると思われる動作をできるだけ試してその結果を吟味する。
- （注意すべき挙動や見分け方）
