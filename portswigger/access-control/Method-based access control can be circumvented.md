# Lab: Method-based access control can be circumvented

## 概要
このLabではHTTPメソッドに依存した不完全なアクセス制御を扱う。  
目的はこのHTTPメソッドのアクセス制御の脆弱性を理解し、攻撃手法を考え、Burpで試すことである。

---

## 脆弱性の種類
Access Control

---

## 攻撃の結論（最短）
権限昇格のリクエストのパラメータのセッションキーを書き換え、BurpでリクエストのPOSTメソッドをGETメソッドに変える。

## 学んだこと
- （このLabで理解できたこと）
- アクセス制御はHTTPメソッドとURLの組み合わせでされることが多い。
- フロントエンド（門番）は一次フィルターでリクエストがバックエンド（受付）に届いたらこのセッションIDの持ち主は本当にこの処理を許されている
- 管理者権限を持つのかをデータベース等でIF文チェックを行う必要がある。（多層防御）
- ハッカーの定石: 目的のURL（例：/admin-roles）に対して POST でアクセスして「401 Unauthorized」や「403 Forbidden」で弾かれた場合、
- 絶対にそこですぐに諦めない、
- 試すべきこと: メソッドを GET、PUT、PATCH、さらには存在しないデタラメな文字列（POSTX や HACK など）に書き換えて再送信する。門番のルールの「隙間」を縫って、バックエンドにリクエストを届ける（バイパスする）ことができるかを必ずテストする。

## メモ（任意）
- 診断手法の視点：「正常系」から「異常系」を導き出す手順
- 正常系の観察: 高い権限（administrator）を使って、まずはシステムが「正しい状態」でどのようなHTTPリクエスト（URL、メソッド、パラメータの形）を送っているかをBurp Suiteで記録する。
- 権限のダウングレード: 低い権限（wiener）のセッションIDにすり替えて、同じリクエストを送ってみる。
- バイパスの試行: 弾かれたら、リクエストの一部（今回はHTTPメソッド）を改ざんして、防御壁の抜け道を探る。
